# Ian mychouchou 怪诞正文配图

## 核心定位

为中文文章设计和生成 16:9 横版正文配图。目标不是做商业插画、PPT 信息图或可爱卡通，而是把文章里的关键判断、流程、结构、状态或隐喻，变成一张清爽、怪诞、有创意、可读但不说明书的手绘解释图。

默认视觉 IP 是“mychouchou / 臭臭”。保留原有工作流逻辑，但视觉身份已经切换为布偶猫臭臭，具体以 `references/mychouchou-ip.md` 和已确认母版为准。mychouchou 必须参与画面的核心动作，不能只是站在旁边当装饰。

它的一个词识别点是：

**狮偶**

内部判断标准保持不变：先看见它在干活，再看见它可爱。

## 先读这些参考

按任务需要读取，不要一次塞满上下文：

- `references/style-dna.md`：风格 DNA、颜色、文字、禁忌。
- `references/mychouchou-ip.md`：mychouchou IP 的形象、性格、动作库和禁忌。
- `references/composition-patterns.md`：结构类型、原创隐喻方法和反复刻规则。
- `references/prompt-template.md`：单张生图提示词模板。
- `references/qa-checklist.md`：生成后检查和迭代规则。
- `assets/examples/`：只作低频视觉校准，不进入默认生成路径。不要照抄这些案例的构图、物件或标注。

## 工作流

### 1. 消化正文

先读用户给的正文、链接、Notion 页面、Markdown 文件或截图内容。提炼：

- 核心观点是什么
- 哪些段落承担认知转折
- 哪些内容适合用图解释
- 哪些地方只适合文字，不需要图

不要平均配图。优先选择“认知锚点”，例如：核心判断、两个断点、输入输出闭环、分流、前后对比、一鱼多吃、承接路径、常见坑、角色状态变化。

### 2. 先判内容结构，再选画法

不要先想到“这张画成流程图/机器/地图吧”，而是先判断内容逻辑，再决定最合适的视觉骨架。

优先按下面顺序判断：

- 谁包含谁、上下层级很明确：用树状/分层结构。
- 两个概念的交集：用双圆交叠或中间重合物。
- 两个维度交叉分类：用四象限或二维格子。
- 一个大主题下面并列子类：用中心分组、放射分组或云朵分组。
- 有明显先后顺序：用线性流程、路径或接力结构。
- 如果以上都不完全成立，先重写拆解方式，不要硬套老构图。

一张图只允许一个主结构，不混搭多种版型。

### 3. 先出配图策略

如果用户只是说“分析怎么配图 / 思考哪些地方需要配图”，先给 shot list。每张图写清楚：

- 放在哪个段落后
- 图的主题
- 核心意思
- 结构类型
- mychouchou在图里做什么
- 建议元素
- 建议中文标注词

默认 4-8 张。文章很短时 1-3 张；长文也不要轻易超过 9 张。够用就好，避免把正文做成画册。

### 4. 单张生成

如果用户明确要求“生成 / 输出 / 做图 / 帮我生成”，不要停下来等确认；用内置 `image_gen` 每张单独生成。不要把多张图拼在一张里。

每张图只讲一个核心结构。提示词必须包含：

- 16:9 横版中文正文配图
- 纯白背景
- 黑色手绘线稿
- 少量红色/橙色/蓝色中文手写批注
- 大量留白
- mychouchou作为核心动作主体
- 禁止 PPT、商业插画、幼稚可爱、复杂架构、左上角类型标题

mychouchou 的识别锚点必须稳定，并以已确认母版为准：

- 布偶猫圆脸、大圆蓝眼、奶灰白双色面罩必须稳定。
- 白色中轴脸、白口鼻和白脸颊要稳定，不要画成别的猫脸花色。
- 身体必须是中长毛布偶猫的蓬松躯干，不要变成短毛猫身体。
- 毛量要有一点小狮子般的威风感，但不能炸毛，不能毛球化。
- 腿是中等长度：不是短腿胖团，也不是长腿纤细挂。
- 神态要乖、无辜、软糯，但工作状态下可以专注、思考或轻微臭脸。
- 内部标准永远是：先看见它在干活，再看见它可爱。

如果用户已经确认过一张mychouchou脸部母版，后续生成时优先服从那张母版的人脸一致性。先对脸，再对动作，再对结构。

默认优先生成“少字版”或“无标题版”画面：

- 不让模型生成长中文段落。
- 不让模型生成左上角类型标题。
- 标注词能少就少，通常 3-6 个短词足够。
- 如果用户要做严肃信息图，优先让模型生成角色、图示、路径和色块骨架，再由排版工具后加文字。
- 默认优先卡通化、简化、平面化的手绘布偶猫，不要往写实宠物插画方向漂。

不要复刻过往案例。案例只提供风格密度和mychouchou参与方式，不能直接复用“传送带断点 / mychouchou拉线 / 素材鱼 / 盖章工具箱 / 常见坑路径”等已有构图，除非用户明确要求复刻某张图。每次都要从当前文章重新发明一个奇怪但成立的隐喻。

### 5. 素材复用优先级

如果任务变成系列图，不要每张都从零开始想全部元素，而是优先稳定复用三类东西：

- mychouchou身份锚点：脸、嘴边灰斑、眼神、腿部花色。
- 常用动作库：拉、扛、推、牵线、守门、分拣、记录、校验。
- 低科技物件库：箱体、抽屉、门、闸门、线团、秤、怪机器、井、梯子、水管。

复用的是角色一致性和物件语法，不是旧图构图。

### 6. 检查与迭代

生成后检查 `references/qa-checklist.md`。如果出现以下问题，优先重生成或局部编辑：

- mychouchou只是装饰
- 画面太满
- 太像流程图/PPT
- 中文太多或错字严重
- 左上角出现“常见坑/流程图/系统架构图”等标题
- 画风太可爱、幼稚、死板
- 背景不是干净白底
- 嘴边灰斑位置不对，和其他图不一致
- 四条腿都变成白腿
- 看起来像宠物插画，不像工作型 IP

### 7. 保存交付

如果用户在 workspace 内工作，把最终图复制到：

```text
assets/<article-slug>-illustrations/
```

按顺序命名：

```text
01-topic-name.png
02-topic-name.png
```

保留原始生成文件，不要覆盖已有资产，除非用户明确要求替换。

## 输出口径

生成前的策略输出要短而准。生成后的交付要包含：

- 生成了几张
- 每张图的用途
- 保存路径
- 哪些图最稳，哪些图是可选

不要长篇解释风格理论；让图自己说话。
